当前设计基线允许进入技术架构设计。

理由：

- 产品总纲、首发纵切、核心循环、节点地图、经济、遭遇、Build、UI、长期边界、中后期主系统和逻辑架构均已落档。

- 逻辑架构已经冻结 `ActionCommand` 管线、核心状态对象、内容门禁、账本投影、存档迁移和架构验收场景。
- 所有规格文档已统一为 `状态：frozen`，架构输入清单不再保持 `working draft`。

## 通过项

### 1. 首发闭环可以进入技术架构

首发 S0 的 8 回合闭环已经有足够输入：

- 时间/AP 与锚点窗口
- 节点地图与移动代价
- 暴露与安全区
- 元石、功绩、债务、药堂、黑市

- 求活路线与 Build 痕迹
- 账本式 UI 投影
- 阶段存档

### 2. 中后期状态边界未压扁

以下长期对象已独立保留：

- 主修流派 / 道痕
- 蛊虫喂养与维护负担
- 初成福地质量 / 洞天跃迁位
- 仙窍时间流速
- 灾劫周期
- 五域天地二气适配 / 仙窍补气
- 宝黄天接入媒介与神念 / 意志交互
- 杀招 / 阵 / 仙蛊屋
- 多资产容器

### 3. 内容与规则边界明确

现有文档明确：

- 规则事实只由规则模块和策展内容产生。
- 实时生成只负责表达、上下文化和局部变体。
- `canon_strict` 与 `sandbox_if` 共用状态结构，只在内容门禁、锚点容差和 IF 许可度上不同。

### 4. UI 架构方向明确

账本式 UI 已经定义为母界面，而不是视觉皮肤。

后续仙窍、宝黄天、阵、仙蛊屋和组织委托都应展开为账页，不另起系统壳。

## 剩余风险

### 1. 技术选型仍未冻结

当前设计文档不绑定语言、框架、运行时和数据格式。

处理方式：

- 由技术架构设计冻结首发实现技术栈。
- 技术选型不得反向压扁逻辑架构对象。

### 2. 首发数值尚未进入平衡表

已有节奏、压力和验收标准，但还没有最终数值表。

处理方式：

- 首发实现计划只要求最小可玩数值。
- 真正平衡表在可跑通 8 回合后单独建立。

### 3. 内容资产格式尚未确定

已有内容门禁和证据等级，但没有落到具体 schema。

处理方式：

- 技术架构定义内容资产格式。
- 纵切实现计划只建立最小内容包。

## 架构准入条件

技术架构设计必须满足：

- 明确语言、运行时、测试框架、数据格式和存档格式。
- 明确 `RunState`、`TimeState`、`WorldSpaceState`、`BuildState`、`SaveEnvelope` 等对象的技术表达。
- 明确 `ActionCommand` 管线如何被测试。
- 明确 UI 只读账本投影。
- 明确实时生成不能写入规则事实。
- 明确 S0 首发纵切不依赖 S1-S4 完整实现。
