## 文档角色

本文档把“账本式 UI 信息架构”进一步冻结成 Phase 7 可执行的视觉与交互约束。

它回答以下问题：

- Phase 7 为什么不能继续使用临时 demo UI。
- “冷峻账本”在颜色、排版、组件和信息优先级上是什么意思。
- 后续 AI 或前端实现者如何避免把界面做成普通网页 dashboard。
- 哪些 UI 决策必须在 Phase 7 前锁住，哪些仍留给实现打磨。

本文档不定义具体 React 组件代码，不替代 Rust 投影，不新增玩法规则。

## 核心视觉方向

主方向：冷峻账本。

玩家面对的不是现代管理后台，而是一套在青茅山秩序缝隙里求活的账簿：

- 修行账。
- 债务账。
- 因果账。
- 风声簿。
- 节点行迹。
- Build 缺口记录。

界面应给人压迫、克制、低温、利益清算的感觉。

禁止方向：

- 紫色 AI SaaS 界面。
- 泛营销页大渐变。
- 霓虹终端风。
- 纯水墨装饰皮肤。
- 治愈生活手账。

## 参考项目使用边界

公开 GitHub 设计项目只作为方法参考，不作为视觉皮肤来源。

- `google-labs-code/design.md`：参考其“设计 token + rationale + lintable contract”的文档方式。
- `cola-runner/awesome-tui-design`：参考高信息密度、主状态不离眼、键位/操作清晰的经验。
- `hirokaji/jp-ui-contracts`：参考中日韩文本排版、行宽、混排稳定性的约束方式。
- `VoltAgent/awesome-design-md`：只参考其反 AI 平庸前端的整理方向，不套用网页产品模板。
- `VoltAgent/awesome-claude-design`：只参考 anti-slop 提醒，不复制任何品牌或示例风格。

## Phase 7 UI 固定结构

### 顶层状态条

必须常驻显示：

- 当前日。
- 当前时段。
- 窗口类型。
- AP。
- 当前节点。
- 暴露。
- 债务压力。
- 伤势。
- 当前遭遇。

顶层状态条是生存仪表，不是标题栏。

### 主账页

Phase 7 至少要组织以下账页：

- 正文场景页。
- 因果账本页。
- 节点地图页。
- 物资与债务页。
- 关系局势页占位。
- 空窍 / 修行页。
- Build 页。
- 风声与线索页。

每一页都应像同一本账簿的不同页，不允许各自变成独立 dashboard 卡片。

### 危机前置信息

以下状态必须主动前置：

- 暴露升高。
- 药堂债、人情债或组织债进入追索。
- 伤势压缩 AP。
- 活跃遭遇阻断普通行动。
- 黑市或传承入口改变风险。
- 路线缺口造成选项关闭。

## 信息表达规则

### 世界内包装

系统信息要包装成世界内账册语言。

推荐：

- “风声渐紧”。
- “药堂账未清”。
- “伤势牵扯行动”。
- “此路暂不可行”。

避免：

- “riskScore +10”。
- “routeFlag disabled”。
- “relationshipModifier changed”。
- “quest failed”。

数值可以显示，但必须与世界内解释并存。

### 稳定术语

同一状态要长期使用稳定词汇：

- 暴露。
- 债务压力。
- 风声。
- 伤势。
- 求活路线。
- 主修流派。
- 核心蛊。
- Build 缺口。

不要在不同页面随意换名。

### 中英文混排

中文标签优先。英文 ID 只作为二级调试信息。

可接受：

- “当前节点：学堂门口 `academy_gate`”

不可接受：

- “Current Node: academy_gate” 作为主显示。

## 组件冻结

### 行动按钮

行动按钮只负责提交 `ActionCommand`。

按钮可以显示：

- 行动名称。
- 可见代价。
- 风险提示。
- 不可用原因。

按钮不可：

- 在 React 中直接扣资源。
- 在 React 中直接改节点。
- 在 React 中生成规则奖励。
- 调用 AI 生成正文后再提交行动。

### 账本行

重复状态优先使用账本行：

- 标签。
- 可见值。
- 最近变化。
- 风险标记。

账本行比普通卡片更适合资源、债务、伤势、关系和 Build 缺口。

### 压力标记

压力标记分三类：

- 普通压力：青灰或暗金。
- 高风险：朱砂。
- 不可逆或接近断裂：朱砂加深并前置。

颜色不能成为唯一信息。文字也必须说明原因。

## Phase 7 与当前 UI 的关系

当前 Sprint 0 UI 是功能壳，不是最终风格基线。

Phase 7 应顺手修正：

- 中文源码乱码。
- 临时 hero 文案。
- demo 式卡片堆叠。
- 状态信息散落。
- Build 页信息不足。
- 遭遇 / 伤势信息与行动按钮分离。

但 Phase 7 不应做：

- 完整美术资产。
- 完整动画系统。
- 多主题系统。
- Web 试玩壳。
- 新玩法规则。

## 与长期系统兼容

未来新增系统仍进入账本母体：

- 仙窍经营：空间经营账页。
- 宝黄天交易：交易账页。
- 杀招：Build 战斗层账页。
- 阵：场域部署账页。
- 仙蛊屋：载体与维护账页。

如果中后期必须换一套 UI 壳，说明 Phase 7 风格冻结失败。

## 验收标准

- 玩家能在主界面上始终看见时段、窗口、AP、节点、暴露、债务、伤势和遭遇压力。
- 玩家能在 1-2 次切换内找到地图、债务、关系、Build、线索。
- 所有页面像同一本账，而不是不同系统拼贴。
- 中文文本无乱码。
- UI 不读取隐藏变量原值。
- UI 只提交 `ActionCommand`。
- UI 不调用 AI 生成正文。
- 视觉上避开紫色 SaaS、营销渐变、纯终端 UI 和纯水墨皮肤。

## 失败信号

- 账本只是做旧背景，没有统一信息组织。
- 玩家必须跨很多层才能看到最紧压力。
- Build、债务、地图、线索各用一套语言。
- 行动按钮把规则效果写在 React。
- 中后期仙窍、交易、阵、仙蛊屋无法自然成为新账页。

## 本命蛊 UI 补冻

Phase 7 的 Build / 修行账页必须显示 `本命蛊：未建立`，但不得暗示首发 S0 已开放本命蛊获取、绑定、改命或替换玩法。

UI 展示必须区分：

- `核心蛊`：当前 Build、杀招或冲突决策的中心。
- `辅助蛊`：支撑核心蛊或补足行动缺口。
- `本命蛊`：生命 / 空窍绑定状态位，首发只显示未建立。

禁止把本命蛊做成普通装备槽、技能槽、职业天赋或默认等同于核心蛊。
