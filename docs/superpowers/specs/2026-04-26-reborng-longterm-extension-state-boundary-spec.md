## 文档角色

本文档不是玩法文档，而是架构前的公共契约文档。它一次性锁住长期扩展边界，回答以下问题：

- 哪些状态是首发就要按长期眼光命名和分层的
- 哪些东西属于凡人层，哪些必须留给蛊仙层
- 世界空间、Build、资源、资产所有权和存档状态如何分层
- 什么样的设计会把未来逼进重构坑

## 实现前公共接口问题

- 该系统的主循环是什么，最小操作单位是什么
- 它消耗哪些：时间、AP、资源、暴露、债务、人情、空间占用
- 它产出哪些：成长、材料、资格、部署能力、交易能力、关系变化
- 它依赖哪些上位状态：修为、空间、节点、路线、Build、资产所有权
- 它的失败形态是什么：受损、拖慢、欠账、暴露、路线断裂、资产损失
- 它在 `canon_strict` 与 `sandbox_if` 下哪里不同
- 它要在账本 UI 上最少显示哪些信息
- 它如何与首发系统兼容，而不是推翻首发系统

## 核心原则

### 1. 首发不是试验田

- 首发范围可以小
- 但首发状态命名与结构不能短视

### 2. 分层优先于细化

- 在所有远期系统写细前，必须先把状态层级分清
- 分不清层级，比没写细更危险

### 3. 不用一个字段承受多个阶段语义

- 凡人期有效的字段，不应硬承担蛊仙期和高阶部署期语义
- 一旦一个字段同时承担三层以上语义，未来必重构

## 修为阶段分层

本项目长期至少分为以下层：

- `凡人层`
- `升仙断层`
- `蛊仙经营层`
- `高阶部署层`
- `尊者级 IF 层`

### 凡人层

- 主要围绕元石、修行、债务、黑市门路、传承残线、制度压迫

### 升仙断层

- 不是简单升级
- 是角色状态、空间结构、资源逻辑与风险结构的一次换挡

### 蛊仙经营层

- 进入仙窍产出、仙元、仙材、跨图与交易信用

### 高阶部署层

- 杀招、阵、仙蛊屋、组织资源、跨域任务与大体量资产开始形成

### 尊者级 IF 层

- 是极后期高自由博弈层
- 不能反向决定首发字段写法

## 世界空间分层

世界空间至少分五层：

- `公开主地图空间`
- `边缘与隐秘接触点`
- `跨图区与地区切换空间`
- `仙窍内空间`
- `交易与部署空间`

### 公开主地图空间

- 家族、学堂、功绩、药堂、酒肆、山道等首发节点

### 边缘与隐秘接触点

- 黑市接入口
- 传承残线
- 灰路关系点

### 跨图区与地区切换空间

- 解决玩家如何离开青茅山、进入别的地区、跨势力移动

### 仙窍内空间

- 不是普通地图子节点
- 是独立经营、产出、承载与部署空间

### 交易与部署空间

- 宝黄天挂单
- 仙蛊屋部署位
- 阵地布置位

## 资源层分层

资源至少分三层：

- `凡人层资源`
- `蛊仙层资源`
- `部署与组织层资源`

### 凡人层资源

- 元石
- 材料
- 功绩
- 债务
- 人情
- 黑市门路

### 蛊仙层资源

- 仙元
- 仙材
- 仙窍产出
- 交易信用
- 宝黄天挂单能力

### 部署与组织层资源

- 阵位
- 仙蛊屋部件位
- 组织供给
- 外部托管资产

禁止：

- 用“通用货币”抹平所有层
- 用“通用信誉”混写债务、人情和交易信用

## Build 分层

Build 至少分五层：

- `凡人求活层`
- `蛊组合层`
- `杀招层`
- `场域部署层`
- `经营支撑层`

### 凡人求活层

- 路线倾向
- 债务关系
- 资源来源
- 求活风格

### 蛊组合层

- 核心蛊
- 辅助蛊
- 组合缺口

### 杀招层

- 杀招模板
- 激活条件
- 资源消耗

### 场域部署层

- 阵
- 仙蛊屋
- 外部载体

### 经营支撑层

- 仙窍产出
- 组织供养
- 交易链

## 资产所有权分层

为了避免未来重构，资产必须至少能区分以下容器：

- `随身`
- `落脚点`
- `公开库存或组织库`
- `仙窍内`
- `挂单中`
- `部署中`
- `外借或托管中`

同一件东西在不同容器里，语义不同。

例如：

- 随身的蛊可直接参与遭遇
- 挂单中的东西不能同时参加部署
- 仙窍内资产可参与经营，但不一定可立刻取用

## 存档与状态演进

### 1. 必须跨阶段保留的状态

- 路线倾向
- 核心关系与敌意
- 债务痕迹与信誉痕迹
- 主要资源来源偏好
- 核心蛊与 Build 主轴
- 重要伤势或结构性后遗症

### 2. 可以阶段重置或弱化的状态

- 短期窗口 AP
- 局部场景一次性风险标记
- 一次性小道消息热度

### 3. 不能含糊的地方

如果一个状态未来是否保留说不清，就先假设它要跨阶段保留，并在文档中标明。

## 禁止性条款

以下设计视为高危：

- 用一个“当前地图”字段同时表示主地图、仙窍和交易空间
- 用一个“货币”字段同时表示元石、仙元和挂单信用
- 用一个“持有蛊列表”同时表示随身、仙窍、部署和交易占用
- 用一个“Build”页只显示蛊，不显示关系、债务、支撑与部署层

## 架构输入清单的公共栏目

未来正式架构文档必须至少列出以下对象类：

- 修为阶段状态对象
- 世界空间对象
- 资源层对象
- 资产容器对象
- Build 对象
- 债务/信誉对象
- 存档演进对象

这份文档不替代架构设计，但决定架构至少得有哪些大对象。

## 模式差异

### canon_strict

- 分层更受原著证据约束
- 远期高自由内容要更谨慎落地

### sandbox_if

- 容许更早开放部分高风险层
- 但不能破坏分层本身

## 验收标准

- 每个新系统都能明确挂到已有层，而不是悬空
- 凡人层与蛊仙层资源不会被混写
- 主地图、仙窍、交易空间不会被当成同一种地图
- Build 能从首发自然挂接到杀招、阵和仙蛊屋
- 资产能区分随身、经营、挂单和部署状态

## 失败信号

- 新系统一来就要求重命名一大批首发概念
- 一个字段开始承受多个阶段语义
- 宝黄天、黑市、药堂债、交易信用开始互相混写
- 仙窍被当成新地图皮肤，而不是新空间层
