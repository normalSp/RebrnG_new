适用范围：S0 到凡人阶段的蛊虫状态建模、炼化归属、炼蛊 / 合炼 / 升转接口，以及未来杀招组成的底层合同。

## 文档角色

本文档冻结“状态合同级”规则，不冻结完整蛊虫图鉴、完整蛊方、成功率公式、杀招列表或数值平衡表。

它的目的不是提前写完全部玩法，而是防止实现时把蛊虫写成普通装备，把炼蛊写成无风险 crafting，把升转写成 XP 升级，把杀招写成技能树按钮。

## 原著一致性原则

### 1. 蛊是活的核心资产，不是装备栏词条

蛊虫必须同时影响：

- 行动选项。
- 修行路线。
- 资源消耗。
- 喂养维护。
- 风险暴露。
- Build 缺口。
- 关系与债务。

首发可以只开放少量蛊虫例子，例如月光蛊，但底层状态不能退化成“装备 + 数值加成”。

### 2. 炼化归属与炼蛊不是同一个动作

必须区分：

- `炼化归属`：让已有蛊虫归玩家可用，重点是控制权、空窍承载、意志压制、风险与耗时。
- `炼蛊 / 合炼`：按蛊方、材料、环境和条件炼制新蛊或组合结果。
- `升转`：让蛊虫向更高品转变化，必须受修为、材料、蛊方、风险和喂养负担影响。

三者都不能写成普通购买、普通升级或无风险制作。

### 3. 杀招是组合模板，不是技能树

杀招必须建立在蛊虫组合、激发条件和代价上。

杀招可以作为首发决断动作的高阶祖型，但不能在凡人前期直接写成可随意点开的技能面板。

## 蛊虫状态合同

### GuSpec

`GuSpec` 表示一种蛊虫的内容定义。

必须保留：

- `gu_id`：稳定内容 ID。
- `display_name`：中文名。
- `rank_band`：品转范围或当前定义品转。
- `path_tags`：流派或用途标签。
- `role_tags`：核心蛊、辅助蛊、消耗蛊、侦查、移动、治疗、攻伐、防护、经营等。
- `feeding_profile`：喂养维护类型或占位。
- `refinement_profile`：炼化 / 炼蛊 / 升转约束或占位。
- `evidence_level`：证据等级。
- `mode_permission`：`canon_strict` / `sandbox_if` 许可。

首发不要求所有字段都有完整数值，但字段语义必须存在。

### GuInstance

`GuInstance` 表示玩家或世界中一只具体蛊。

必须保留：

- `instance_id`：单只蛊实例 ID。
- `gu_id`：指向 `GuSpec`。
- `rank`：当前品转。
- `owner`：所属者或当前控制者。
- `container`：随身、空窍、落脚点、组织托管、挂单中、部署中等。
- `control_state`：未炼化、炼化中、已炼化、失控、受他人痕迹影响等。
- `condition_state`：完好、受损、残蛊、濒毁等。
- `feeding_state`：稳定、将缺、断供、偏食压力等。
- `build_role`：核心、辅助、临时、缺口替代等。

禁止用一个普通 inventory item 同时承担所有这些语义。

## 炼化归属合同

炼化归属的主循环：

1. 确认蛊虫来源和当前控制状态。
2. 检查修为、空窍承载、时间窗口和场所。
3. 支付真元、时间、材料、风险或关系代价。
4. 结算控制权变化、伤势、暴露、债务和账本记录。

可见消耗：

- AP 或窗口。
- 真元 / 元石 / 材料。
- 安全场所。
- 关系掩护。
- 暴露风险。

失败形态：

- 炼化中断。
- 蛊虫受损。
- 真元或资源损失。
- 伤势加重。
- 底牌暴露。
- 欠下药堂债或人情债。

首发可只做占位和账本表达，不必开放完整炼化操作。

## 炼蛊 / 合炼合同

炼蛊 / 合炼的主循环：

1. 获得蛊方或残方。
2. 验真蛊方来源。
3. 准备材料、场所、时间和风险承受。
4. 执行炼制。
5. 结算成品、残蛊、材料损失、伤势、暴露或债务。

必须依赖：

- 蛊方证据等级。
- 材料来源。
- 场所安全。
- 时间窗口。
- 修为阶段。
- 相关蛊虫或辅助条件。

失败形态：

- 材料白耗。
- 产出残蛊。
- 蛊虫损毁。
- 受伤或中毒。
- 被他人盯上。
- 债务或人情加重。

禁止：

- 一键 crafting。
- 无风险合成。
- 缺蛊方也能稳定炼出关键蛊。
- `canon_strict` 下用缺证据原创蛊方产出关键奖励。

## 升转合同

升转是蛊虫品转提升或结构性变化，不是 XP 升级。

升转必须受以下条件约束：

- 玩家修为阶段。
- 空窍 / 仙窍承载。
- 目标蛊当前状态。
- 材料和蛊方。
- 时间窗口。
- 场所风险。
- 喂养维护负担变化。
- 失败后的资产损失。

升转产出：

- 更高品转蛊虫。
- 更重喂养压力。
- 更高暴露价值。
- Build 主轴更清晰。
- 资源来源需求改变。

失败形态：

- 升转失败但蛊虫保留。
- 蛊虫降损或残缺。
- 材料损耗。
- 伤势或反噬。
- 引来争夺、审查或债务追索。

首发只保留升转接口，不开放完整升转系统。

## 杀招组成合同

杀招模板至少包含：

- `killer_move_id`。
- `display_name`。
- `core_gu_requirements`：核心蛊要求。
- `support_gu_requirements`：辅助蛊或替代位。
- `path_alignment`：流派 / 道痕兼容占位。
- `activation_context`：时段、节点、遭遇、距离、伤势或场域条件。
- `resource_cost`：真元、材料、AP / 窗口、暴露或维护代价。
- `failure_consequence`：反噬、资源白耗、Build 暴露、伤势或局面恶化。
- `evidence_level`。
- `mode_permission`。

首发决断动作如跑路、硬顶、拖延、嫁祸，可以被视为未来杀招的低阶行动祖型。

但凡人前期不应让玩家随意拥有完整杀招库。

## 与 Build 的关系

蛊虫、炼化、升转与杀招必须挂入同一个 Build 母体：

- 求活路线层：玩家靠什么活。
- 主修流派 / 道痕层：长期主轴和兼修压力。
- 蛊组合层：核心蛊、辅助蛊、缺口。
- 维持层：喂养维护和供养链。
- 战斗层：杀招模板和冲突风格。
- 关系债务层：谁提供蛊、材料、场所或保护。

禁止把“求活路线”“主修流派”“核心蛊”“喂养维护”塞进一个模糊字段。

## 模式差异

### canon_strict

- 具体蛊名、蛊方、杀招和关键奖励必须来自明确证据或高可信推断。
- 缺证据内容不能产出关键蛊、关键杀招或主线级资源结果。
- 月光蛊等首发锚点可作为明确例子，但不因此展开完整蛊虫表。

### sandbox_if

- 可以使用原创补完蛊虫、残方、杀招雏形。
- 必须显式标记模式许可和证据等级。
- 仍受修为、资源、场所、时间、风险和主线保护门禁限制。
- 不能用 IF 许可抹掉炼化、炼蛊、升转和喂养维护的代价。

## 账本 UI 最低显示

Build 或修行相关账页至少要能显示：

- 当前核心蛊。
- 辅助蛊或缺口。
- 蛊虫状态：完好、受损、残蛊、炼化中等。
- 喂养维护压力。
- 当前求活路线。
- 主修流派保留位。
- 可见炼化 / 炼蛊 / 升转门槛。
- 当前无法行动的原因。

UI 不需要显示隐藏公式，但必须显示玩家能理解的风险原因。

## 架构接口要求

后续 Rust 类型设计不得把以下概念合并：

- `GuSpec` 与 `GuInstance`。
- 炼化归属与炼蛊。
- 炼蛊与升转。
- 蛊虫所有权与资产容器。
- 核心蛊与求活路线。
- 杀招模板与普通行动按钮。
- 喂养维护与普通资源余额。

这类合并会导致后续升仙、仙窍、宝黄天、阵和仙蛊屋扩展时重构。

## 验收标准

- 文档明确区分 `GuSpec` 与 `GuInstance`。
- 文档明确区分炼化归属、炼蛊 / 合炼、升转。
- 文档明确写出蛊虫升转不是 XP 升级。
- 文档明确写出炼蛊不是无风险 crafting。
- 文档明确写出杀招不是技能树按钮。
- 文档保留 `核心蛊`、`辅助蛊`、`杀招模板`、`喂养维护`、`canon_strict`、`sandbox_if`。
- 文档不创建完整蛊虫图鉴、完整蛊方、成功率公式或杀招列表。

## 失败信号

- 蛊虫被写成普通装备。
- 炼蛊被写成一键 crafting。
- 升转被写成经验条升级。
- 杀招被写成技能树。
- 喂养维护被当作普通金币消耗。
- `canon_strict` 下缺证据原创蛊方产出关键奖励。

## 本命蛊补冻 v1

`本命蛊` 是独立于 `核心蛊` 的生命 / 空窍绑定状态位，不得在架构或 UI 中与当前战斗核心、路线核心、杀招核心混成同一个字段。

本命蛊状态至少要保留：

- `vital_gu_status`：未建立、未定、已建立、受损等状态占位。
- `vital_gu_instance_id`：若未来存在具体实例，必须指向某个 `GuInstance`，不能只写名称。
- `binding_scope`：凡人空窍、仙窍或更高阶段绑定范围。
- `binding_risk`：反噬、牵连伤势、空窍负担或长期代价占位。
- `evidence_level`。
- `mode_permission`。

首发 S0 固定显示为 `本命蛊：未建立`，只作为状态保留位，不开放获取、绑定、改命或替换玩法。

`核心蛊` 可以是当前 Build / 杀招 / 冲突决策的中心；`本命蛊` 是更深的生命绑定关系。二者未来可以重合，但必须由显式状态关系表达，不能默认等同。

`canon_strict` 下不得用缺证据原创事件临时建立本命蛊；`sandbox_if` 即使允许原创补完，也必须经过修为、空窍 / 仙窍、风险、证据标签和主线保护门禁。

后续 Rust 类型与 UI 投影不得合并以下概念：

- 核心蛊与本命蛊。
- 本命蛊与普通装备槽。
- 本命蛊与杀招模板按钮。
- 本命蛊与求活路线标签。
