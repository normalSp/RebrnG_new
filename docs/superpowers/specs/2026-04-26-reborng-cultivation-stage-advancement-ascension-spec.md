## 文档角色

本文档负责定义从凡人到升仙的修为推进结构，以及升仙作为系统断层的玩法含义，回答以下问题：

- 修为推进在本作里是怎样的阶段跃迁
- 什么叫凡人期，什么叫升仙断层
- 升仙前后有哪些状态必须变，哪些必须延续
- 为什么升仙不能只做成“等级 +1”

## 实现前公共接口问题

- 该系统的主循环是什么，最小操作单位是什么
- 它消耗哪些：时间、AP、资源、暴露、债务、人情、空间占用
- 它产出哪些：成长、材料、资格、部署能力、交易能力、关系变化
- 它依赖哪些上位状态：修为、空间、节点、路线、Build、资产所有权
- 它的失败形态是什么：受损、拖慢、欠账、暴露、路线断裂、资产损失
- 它在 `canon_strict` 与 `sandbox_if` 下哪里不同
- 它要在账本 UI 上最少显示哪些信息
- 它如何与首发系统兼容，而不是推翻首发系统

## 设计目标

### 1. 修为推进必须是结构变化，不是纯数值抬升

- 低阶与高阶不能只是面板更大
- 每个阶段都要带来资源结构、风险结构或空间结构变化

### 2. 升仙必须是断层

- 升仙不是平滑升级
- 它必须改变世界空间、资源层、Build 层和长期目标

### 3. 首发与未来必须同轴

- 首发凡人期写法必须能自然延伸到升仙
- 不能首发一套、升仙后再换一套世界观规则

## 修为层级总览

长期修为推进建议至少分为以下层：

- `凡人早期`
- `凡人中后期`
- `升仙断层`
- `蛊仙经营期`
- `高阶部署期`
- `尊者级 IF`

## 凡人早期

对应首发：

- 开窍前后
- 一转根基初立
- 靠元石、功绩、药堂、关系、门路和少量核心蛊求活

主循环特点：

- 时间窗口紧
- 地图小
- 伤势重
- 资源稀

## 凡人中后期

对应未来 `S1`：

- 可跨更大地区
- 可接触更重的组织关系
- 可形成更稳定的资源链
- 修为推进不再只围绕学堂与青茅山

主循环特点：

- 路线差异更重
- 势力依附更明显
- Build 开始需要更完整支撑

## 升仙断层

### 1. 升仙不是自动过场

升仙必须是一个完整系统事件，而不是剧情过渡动画。

### 2. 升仙的最低含义

一旦升仙，至少以下几层结构发生变化：

- 资源层：元石逻辑不再够用
- 空间层：仙窍出现
- Build 层：开始需要经营支撑
- 世界层：公开地图不再是唯一核心空间
- 风险层：失败不再只是受伤和欠债，而会出现更大资产代价

### 3. 升仙必须保留的前史

升仙后不能清零：

- 债务痕迹
- 重要关系
- 路线倾向
- 核心 Build 方向
- 关键伤痕与信誉

## 修为推进主循环

### 1. 凡人期主循环

- 修行
- 求活
- 还债或拖债
- 争资源
- 形成 Build 倾向

### 2. 升仙前准备循环

- 攒资格
- 补核心资源
- 修补 Build 缺口
- 处理过高暴露或过重拖累
- 准备承担升仙失败代价

### 3. 升仙后主循环

- 经营仙窍
- 维持核心消耗
- 形成长期资源链
- 承接交易与部署系统

## 修为推进输入

修为推进至少受以下因素共同影响：

- 当前修为阶段
- 当前核心蛊与 Build 结构
- 当前资源层状态
- 当前空间可达性
- 当前伤势与长期后遗症
- 当前关系网络与势力依附

## 修为推进产出

修为推进不只是“面板更强”，还应产出：

- 更高行动资格
- 更大可达空间
- 更重维护负担
- 更复杂 Build 层
- 更长期的资产和部署能力

## 升仙条件语义

本文档先冻结条件语义，不钉死最终配方或数值。

升仙至少要满足三类准备中的多数：

- `修为准备`
- `资源准备`
- `结构准备`

### 修为准备

- 当前主轴不能散
- 需要达到凡人期足够稳定的根基

### 资源准备

- 不是只看钱
- 还要看材料、门路、补救手段和后手空间

### 结构准备

- 当前关系、债务、伤势和路线断裂不能处于极度崩坏态
- 否则就算强行升，也更可能掉进坏结果

## 升仙失败形态

升仙失败不应只写成“死”或“没升上”。

更合适的失败形态包括：

- 结构性重创
- 资源层塌陷
- 路线主轴被迫改写
- 长期负担变重
- 关系或势力代价压身

直接死亡可以存在，但不应是唯一表现。

## 升仙后的新负担

升仙后至少新增以下负担：

- 仙窍维护
- 新资源层压力
- 更大空间下的更大暴露和追索
- 更复杂的 Build 支撑需求

这能防止升仙变成单向爽点。

## 与首发系统的兼容

### 1. 时间/AP 不被废弃

- 升仙后仍然保留时间窗口逻辑
- 但会新增经营子循环与更大时间单位

### 2. 地图系统不被废弃

- 升仙后仍有主地图与地区空间
- 只是要叠加仙窍空间与交易空间

### 3. Build 定义不被废弃

- 仍由路线、蛊、支撑、关系与资源来源构成
- 只是叠加更高层结构

## 模式差异

### canon_strict

- 升仙条件与代价更受原著逻辑约束
- 不随便开离谱捷径

### sandbox_if

- 允许更危险、更偏门或更早的升仙路径
- 但失败后果也必须更重，不可白送

## 账本 UI 最低要求

升仙相关页面至少要能显示：

- 当前修为阶段
- 当前推进状态
- 升仙准备中的主要缺口
- 升仙后预计新增的负担
- 现有前史中哪些会被带过去

## 验收标准

- 修为推进被定义成阶段变化，而非纯数值增长
- 升仙被定义成系统断层，而非升级动画
- 升仙前后的资源层、空间层、Build 层变化清楚
- 首发凡人期文档与升仙规格没有逻辑断裂

## 失败信号

- 升仙只增加面板，不增加新负担
- 升仙后首发债务、关系、Build 痕迹全部清零
- 凡人期和蛊仙期被写成两款不同游戏
