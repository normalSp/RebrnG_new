## 文档角色

本文档负责定义账本式母界面的信息组织方式，回答以下问题：

- 为什么本作不是纯文本页拼接，也不是纯表格管理器
- 正文、节点地图、修行、债务、关系、Build、线索如何归到同一账本母体
- 首发 UI 如何为仙窍、宝黄天、阵与仙蛊屋预留扩展页而不换哲学

## 实现前公共接口问题

- 该系统的主循环是什么，最小操作单位是什么
- 它消耗哪些：时间、AP、资源、暴露、债务、人情、空间占用
- 它产出哪些：成长、材料、资格、部署能力、交易能力、关系变化
- 它依赖哪些上位状态：修为、空间、节点、路线、Build、资产所有权
- 它的失败形态是什么：受损、拖慢、欠账、暴露、路线断裂、资产损失
- 它在 `canon_strict` 与 `sandbox_if` 下哪里不同
- 它要在账本 UI 上最少显示哪些信息
- 它如何与首发系统兼容，而不是推翻首发系统

## 设计目标

### 1. 账本是母界面，不是主题皮肤

- “账本式”不是做旧纸背景
- 它是全游戏统一的信息组织法

### 2. 信息密，但不乱

- 玩家应感到自己在翻修行账册、因果账本、风声簿
- 不应感到自己在切十几个互不相干的系统页

### 3. 主循环信息必须始终在视野内

至少要持续可感知：

- 当前时段
- 当前窗口类型
- 当前 AP
- 当前节点
- 当前暴露或风紧状态
- 当前债务追索压力

### 4. 扩展页必须服从同一母体

- 未来仙窍、交易、阵与仙蛊屋只能展开成账页或子页
- 不能另起一套完全不同的系统壳

## 母界面定义

本作 UI 的母体是 `主账页 + 侧账页 + 条目展开页`。

### 主账页

负责：

- 当前正文场景
- 当前可选行动
- 当前主循环状态

### 侧账页

负责：

- 地图
- 关系
- 物资债务
- 修行与 Build
- 风声线索

### 条目展开页

负责：

- 查看细节
- 回看原因
- 理解某一项状态为何变化

## 首发固定页面组

首发账本至少应固定包含以下页面组：

- 正文场景页
- 因果账本页
- 节点地图页
- 物资与债务页
- 关系局势页
- 空窍 / 修行页
- Build 页
- 风声与线索页

## 页面职责

### 1. 正文场景页

必须承载：

- 当前场景叙述
- 当前窗口类型
- 当前行动入口
- 当前最紧迫风险

### 2. 因果账本页

必须回答：

- 我为什么会变成现在这样
- 最近哪些选择抬高了暴露、债务、伤势或门路

### 3. 节点地图页

必须回答：

- 我现在在哪
- 现在能去哪里
- 哪条路更显眼、哪条路更风紧

### 4. 物资与债务页

必须回答：

- 我还剩什么
- 我欠谁什么
- 什么东西正在逼我下一回合处理

### 5. 关系局势页

必须回答：

- 谁在保我
- 谁在盯我
- 谁可被利用
- 谁变得更危险

### 6. 空窍 / 修行页

必须回答：

- 我现在修行处于什么阶段
- 这轮修行投入意味着什么放弃

### 7. Build 页

必须回答：

- 我更像哪条路线
- 我靠什么活
- 我的核心缺口在哪

### 8. 风声与线索页

必须回答：

- 哪些只是传闻
- 哪些已经验真
- 哪些线索值得冒险，哪些只是诱饵

## 固定顶层状态条

无论玩家停在哪一页，顶层至少应持续可见：

- 章节日
- 可见时段
- 当前窗口类型
- 当前 AP
- 当前节点
- 当前暴露区间
- 当前伤势压力
- 当前最紧的债务或锚点提醒

这条信息不应深藏到次级页里。

## 页面切换原则

### 1. 主循环中优先浅切换

- 常用页应在一跳内到达
- 不应为了看债务再点三层菜单

### 2. 危机信息优先浮出

当出现以下情况时，相关账页信息应主动前置：

- 暴露升高
- 债务进入追索
- 伤势压窗
- 黑市门路被解锁
- 锚点即将落下

### 3. 展开页只看细节，不抢主循环

- 条目展开页服务理解
- 不应劫持主操作流

## 信息表达规则

### 1. 用世界内包装，不用纯后台术语

例如：

- 暴露可以写成“露痕”“起疑”“风紧”
- 线索可以写成“风声”“口风”“旧闻”
- 债务可以写成“记账”“欠口”“追索”

### 2. 但不能为包装牺牲可读性

- 玩家仍要清楚知道什么在变糟
- 包装不能把关键风险藏没

### 3. 同类信息表达必须稳定

- 同一个状态不要今天叫“风紧”，明天又变成别的系统词
- 术语一旦定下，就贯穿首发与中后期

## 首发布局建议

首发推荐采用：

- 中央正文区
- 右侧状态摘要区
- 左侧可收合账页入口
- 底部当前行动与代价区

这样可以保证：

- 正文不被挤成边栏
- 主循环状态不离眼
- 账页切换仍有“翻账本”的感觉

## 扩展兼容条款

未来新增系统必须归入账本母体：

- 仙窍经营：作为空间经营账页展开
- 宝黄天交易：作为交易账页展开
- 阵与仙蛊屋：作为部署账页展开
- 组织委托：作为关系/委托账页展开

不允许出现：

- 首发是一套账本 UI
- 中后期突然切成完全不同的经营游戏壳

## 模式差异

### canon_strict

- 信息提示更谨慎
- 缺证据内容不提前做成明确说明
- 传闻与验真状态区分更硬

### sandbox_if

- 可更早出现灰路提示
- 可更早出现危险门路线索
- 但仍必须区分“传闻”与“已知事实”

## 验收标准

- 玩家能在主界面上始终看见当前时间/AP/节点/暴露/债务压力
- 玩家能在 1-2 次切换内找到地图、债务、关系、Build、线索
- 玩家能感到所有页都属于同一本账，而不是八个系统拼贴
- 未来仙窍、交易、部署型系统能自然想象成新的账页，而不是另一款 UI

## 失败信号

- 账本只剩皮肤，没有统一信息组织
- 玩家要跨很多层才能看到当前最紧的压力
- Build、债务、地图、线索各讲各的语言
- 中后期一扩展就必须整套换 UI
